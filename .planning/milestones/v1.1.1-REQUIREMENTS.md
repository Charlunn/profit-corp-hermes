## Requirement Outcomes

### v1.1.1 Requirements

#### GitHub Authentication and Targeting

- [x] **GHAUTH-01**: Approved-project delivery can treat an already-authenticated `gh` CLI session as a valid GitHub auth source when explicit `GH_TOKEN` / `GITHUB_TOKEN` env vars are absent  
  **Outcome:** Validated in Phase 14
- [x] **GHAUTH-02**: Approved-project delivery can still use explicit `GH_TOKEN` / `GITHUB_TOKEN` env vars when they are intentionally provided for non-interactive runs  
  **Outcome:** Validated in Phase 14
- [x] **GHOWN-01**: New approved projects derive the GitHub repository owner from an operator-controlled source instead of falling back to the project slug as the owner  
  **Outcome:** Validated in Phase 14
- [x] **GHOWN-02**: New approved projects derive the repository name and canonical repository URL consistently for both create and attach flows  
  **Outcome:** Validated in Phase 14

#### GitHub Sync Reliability

- [x] **GHSYNC-01**: GitHub sync excludes generated dependency/build directories such as `node_modules`, `.next`, `dist`, and other non-source snapshots from the canonical workspace commit  
  **Outcome:** Validated in Phase 15
- [x] **GHSYNC-02**: GitHub sync succeeds on the real Windows + pnpm workspace shape used by live approved-project delivery without failing on long paths  
  **Outcome:** Validated in Phase 15
- [x] **GHSYNC-03**: GitHub sync can converge the configured remote when the workspace remote does not match the approved project repository  
  **Outcome:** Validated in Phase 15
- [x] **GHSYNC-04**: GitHub sync can use a working Git transport strategy for the operator environment instead of failing solely because HTTPS push is unavailable while SSH is healthy  
  **Outcome:** Validated in Phase 15
- [x] **GHSYNC-05**: GitHub sync writes evidence that distinguishes stage failures caused by staging, commit, remote, or push behavior  
  **Outcome:** Validated in Phase 15

#### Vercel Delivery Reliability

- [x] **VERC-01**: Vercel linkage and deployment can use an available local authenticated Vercel CLI session when explicit `VERCEL_TOKEN` is absent but operator login is already valid  
  **Outcome:** Validated in Phase 16
- [x] **VERC-02**: Vercel linkage and deployment can still use explicit `VERCEL_TOKEN` for non-interactive automation runs  
  **Outcome:** Validated in Phase 16
- [x] **VERC-03**: Vercel deploy records the real linked project name, scope, deploy URL, and deployment evidence for the approved project instead of stale values from previous runs  
  **Outcome:** Validated in Phase 16
- [x] **VERC-04**: Vercel delivery failures distinguish invalid token, inaccessible scope, missing CLI, and actual deployment failure as separate operator-visible outcomes  
  **Outcome:** Validated in Phase 16

#### Authority and Status Convergence

- [x] **STAT-01**: After GitHub sync succeeds, the authority record converges to the correct repo metadata, sync status, commit, and resume stage  
  **Outcome:** Validated in Phase 17
- [x] **STAT-02**: After Vercel link and deploy succeed, the authority record converges to the correct Vercel project metadata, deploy URL, deployment status, and evidence paths  
  **Outcome:** Validated in Phase 17
- [x] **STAT-03**: `DELIVERY_PIPELINE_STATUS.md` renders the latest true final state from authority data and event history instead of continuing to show a stale blocked view after successful recovery  
  **Outcome:** Validated in Phase 17
- [x] **STAT-04**: Operator-facing status surfaces clearly separate historical failures from the current authoritative state so a successfully recovered delivery run reads as completed, not still blocked  
  **Outcome:** Validated in Phase 17

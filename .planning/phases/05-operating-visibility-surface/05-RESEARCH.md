### Locked Decisions
### Freshness and alerts
- **D-01:** The visibility surface should stay calm by default and only elevate exceptions when there is a meaningful issue.
- **D-02:** Blocking states such as governance `blocked`, pending approvals, failed source collection, or clearly stale daily artifacts must be promoted to the top alert area.

- **D-04:** The operator-facing summary should show only the Top 3 next actions.
- **D-05:** The action list should optimize for solo-operator focus rather than acting like a task board or broad work queue.
- **D-06:** Each surfaced action should remain tied back to trusted evidence or governance state instead of becoming a detached recommendation list.

## Practical Status Heuristics

### Recommended severity model
| Status | Machine-checkable trigger | Operator treatment |
|-------|----------------------------|--------------------|
| `action-required` | Any `## Active Blocks` entry other than `- None`, any `## Pending Approvals` entry other than `- None`, or `failed_source_count > 0`. [VERIFIED: repo code][VERIFIED: context] | Render a top alerts section before the normal summary. [VERIFIED: context] |
| `stale` | `completed_at` from `LATEST_SUMMARY.md` is older than 36 hours relative to render time. [ASSUMED] | Render a freshness warning in the top section and downgrade calm mode. [ASSUMED] |
| `watch` | External intelligence is fresh, but the operating package date does not match the latest summary date or governance has recent override entries without active blocks. [VERIFIED: repo code][ASSUMED] | Render a compact “watch” note, not a full alert block. [ASSUMED] |
| `healthy` | No blocks, no pending approvals, `failed_source_count = 0`, and latest-summary freshness inside threshold. [VERIFIED: repo code][ASSUMED] | Keep the file short and compact. [VERIFIED: context] |

### Concrete parsing heuristics
- Treat governance as blocked/pending based on section contents in `GOVERNANCE_STATUS.md`, not by scanning prose elsewhere. [VERIFIED: repo code]
- Treat source collection failure as present when `LATEST_SUMMARY.md` shows `failed_source_count` greater than `0` or `failed_sources` not equal to `none`. [VERIFIED: repo code]
- Treat top actions as valid only when they come from `## 推荐下一步` in the operating package or from explicit pending/blocked governance lines. [VERIFIED: repo code][VERIFIED: context]
- Cap surfaced actions at 3 after merging decision-package actions with urgent governance actions, preferring governance-unblock actions first when exceptions exist. [VERIFIED: context][ASSUMED]
- If the operating package exposes fewer than 3 actions, preserve the smaller count instead of padding. [VERIFIED: repo code]

## Common Pitfalls

### Pitfall 1: Letting the visibility file become a second decision package
**What goes wrong:** The latest view starts adding new narratives, extra strategy, or work-queue content beyond the operating package. [VERIFIED: context]
**Why it happens:** Visibility feels like a place to “improve” the underlying summary instead of presenting it. [ASSUMED]
**How to avoid:** Keep the operating package as the primary anchor and limit visibility to summarization plus overlays. [VERIFIED: context]
**Warning signs:** The visibility artifact contains recommendations that do not appear in `## 推荐下一步` or governance exceptions. [VERIFIED: repo code][VERIFIED: context]

### Pitfall 2: Alert fatigue in healthy runs
**What goes wrong:** Operators see warning-heavy chrome even when nothing is blocked or stale. [VERIFIED: context]
**Why it happens:** Status systems often surface every check equally. [ASSUMED]
**How to avoid:** Only elevate meaningful exceptions and keep healthy mode compact. [VERIFIED: context]
**Warning signs:** The healthy latest view is longer than the board briefing’s first screen. [ASSUMED]

### Pitfall 3: Source-rank inversion
**What goes wrong:** The implementation takes `BOARD_BRIEFING.md` or `EXECUTION_PACKAGE.md` as the main source because they are shorter. [VERIFIED: context]
**Why it happens:** Shorter derivatives are easier to parse than the operating package. [ASSUMED]
**How to avoid:** Always derive current situation, risks, opportunities, and default actions from `OPERATING_DECISION_PACKAGE.md` first. [VERIFIED: context][VERIFIED: repo code]
**Warning signs:** A visibility action exists in the board briefing but not in the operating package. [VERIFIED: repo code][ASSUMED]

### Pitfall 4: Using file timestamps as the only freshness signal
**What goes wrong:** Freshness status becomes unreliable across environments or copy operations. [ASSUMED]
**Why it happens:** File mtimes are easy to inspect. [ASSUMED]
**How to avoid:** Prefer `completed_at`, `run_id`, and failure metadata from `LATEST_SUMMARY.md`, with artifact dates used only as secondary checks. [VERIFIED: repo code][ASSUMED]
**Warning signs:** A file looks recent by mtime, but `LATEST_SUMMARY.md` still shows an old `completed_at` or failed sources. [VERIFIED: repo code][ASSUMED]

## Resolved Planning Choices

1. **Phase 5 will write dated history snapshots by default.**
   - Resolution: The implementation will write both `assets/shared/visibility/OPERATING_VISIBILITY.md` and `assets/shared/visibility/history/{date}-operating-visibility.md`.
   - Why resolved: This matches the repo’s established latest+history pattern for derived artifacts and keeps latest as the operator-default surface.

2. **The stale threshold is one expected daily cycle.**
   - Resolution: Primary artifacts are stale when they are older than one expected daily cycle relative to the latest successful run metadata.
   - Why resolved: The repo’s operating cadence is daily, and the current Phase 5 plans already encode this as the machine-checkable freshness rule.

3. **Urgent governance actions displace decision-package actions only when immediate operator intervention is required.**
   - Resolution: Governance blocked/pending items are always promoted in `## Top Alerts`; they consume a `## Top 3 Next Actions` slot only when the operator must act now to unblock progression.
   - Why resolved: This preserves D-02 alert promotion without violating D-04’s hard cap or unnecessarily displacing evidence-backed operating actions.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| owner approval input → Phase 10 authority script | untrusted/operator-provided approval fields become durable state |
| approved-project authority → downstream bootstrap | top-level record drives workspace creation and must not contain ambiguous/missing data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | S | `start_approved_project_delivery.py` approval input | mitigate | validate required approval fields and stable source identifiers before writing `APPROVED_PROJECT.json` |
| T-10-02 | T | `APPROVED_PROJECT.json` / `PROJECT_BRIEF.md` | mitigate | derive brief only from record fields and lock schema/ordering with unittests so manual drift is detected |
| T-10-03 | R | approval-to-brief transition | mitigate | persist approval evidence, stage, status, and block reason inside the authority record for auditability |
| T-10-04 | I | approved-project bundle paths | accept | artifact paths are repo-local operational metadata, not sensitive secrets, so visibility risk is low |
| T-10-05 | D | missing approval or brief inputs | mitigate | hard-block with explicit `block_reason` and persisted evidence instead of partial writes or silent degradation |
| T-10-06 | E | freeform chat memory as hidden input | mitigate | forbid chat-derived reconstruction and require deterministic projection from the authority record per D-06 |
</threat_model>

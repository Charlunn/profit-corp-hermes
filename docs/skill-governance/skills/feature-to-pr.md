# Skill: feature-to-pr

- 名称：feature-to-pr
- 版本：v0.2
- 维护人：repo owner

## 2. 适用场景
- 从功能需求到可合并 PR 的标准化执行
- 适用于中低风险与可分段交付的功能开发

## 3. 非适用场景
- 生产事故止血（应走 bugfix-safety）
- 纯日志规范整改（应走 logging-convention）

## 4. 输入要求
### 必需输入
- 功能目标（要解决什么问题）
- 验收标准（至少 2 条）
- 约束条件（时间、兼容性、风险边界）

### 可选输入
- 参考页面/API
- 历史方案或竞品

### 缺失输入处理
- 缺失关键验收标准时，不进入实现细节，先补齐需求描述

## 5. 执行步骤
1. 任务拆分：按最小可合并改动拆成 1~3 个子目标
2. 影响评估：标注对 UI/API/DB/权限/日志 的影响面
3. 方案确定：选择最小变更路径，避免无关重构
4. 验证规划：定义主路径与失败路径验证
5. 交付打包：按 Output Contract 生成 PR 内容

## 6. 输出契约（Output Contract）
必须输出以下结构：
- 变更摘要
- 影响面（模块/API/DB）
- 风险点
- 验证证据（主路径 + 失败路径）
- 回滚方案

## 7. DoD（完成定义）
- [ ] 变更范围与目标一致，无额外功能蔓延
- [ ] 硬门禁通过（参考 Playbook）
- [ ] 验收标准逐条有证据对应
- [ ] 回滚步骤可执行

## 8. 反模式与失败案例
- 反模式：需求不清就直接编码
- 风险：返工高、回归高
- 正确替代：先补齐验收标准与边界，再进入实现

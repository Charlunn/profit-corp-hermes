# Skill: logging-convention

- 名称：logging-convention
- 版本：v0.2
- 维护人：repo owner

## 2. 适用场景
- 新增或调整业务流程的日志输出
- 缺陷排查后补齐可观测性

## 3. 非适用场景
- 纯业务功能变更且已有完整日志覆盖
- 无需排查价值的噪声日志堆砌

## 4. 输入要求
### 必需输入
- 业务流程节点（入口、关键动作、外部调用、结束）
- 错误场景定义
- 敏感信息边界

### 可选输入
- 现有日志样例
- 告警/检索需求

### 缺失输入处理
- 无明确排查目标时，不增加日志数量，先定义观测目标

## 5. 执行步骤
1. 标定关键节点：入口、分支、失败、完成
2. 统一字段：按 Playbook 最小字段集合补齐
3. 等级分配：debug/info/warn/error 语义对齐
4. 安全审查：移除敏感数据，保留可定位信息
5. 可用性验证：确保可按 trace_id/request_id 串联链路

## 6. 输出契约（Output Contract）
必须输出以下结构：
- 日志覆盖范围说明
- 字段与等级映射表
- 安全与脱敏说明
- 查询与排查示例
- 回滚方案（如何撤销冗余日志改动）

## 7. DoD（完成定义）
- [ ] 关键路径具备结构化日志
- [ ] 字段、等级与总手册一致
- [ ] 不输出敏感信息
- [ ] 支持按 trace_id/request_id 快速排查

## 8. 反模式与失败案例
- 反模式：只加大量 debug 日志
- 风险：噪声高、成本高、排查反而变慢
- 正确替代：围绕关键节点做少量高价值结构化日志
